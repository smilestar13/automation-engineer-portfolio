# Technical Writing

## Knowledge Sharing

Besides developing automation solutions, I strongly believe that good engineering also includes **sharing knowledge** and producing clear technical documentation.

Throughout my projects I regularly document implementation details, installation procedures, troubleshooting steps and security recommendations so that other engineers can quickly understand and reproduce the solution.

Clear documentation is an essential engineering skill that improves maintainability, onboarding and long-term support of automation systems.

---

## Example Article

One example is my Medium article on Linux server hardening using **SSH configuration** and **Fail2Ban**.

The article has received positive feedback from fellow engineers and was written after a real-world security incident involving one of my colleagues whose server was compromised due to weak SSH protection.

Although the article presents a practical beginner-friendly guide, the security configuration I personally use in production environments is considerably more comprehensive.

My typical server hardening approach includes, among other things:

- Dedicated non-root administrative accounts
- SSH key authentication only
- Disabled password authentication
- Disabled root login where applicable
- Customized SSH configuration
- Fail2Ban protection
- Least-privilege principles
- Additional infrastructure-specific security measures

With this configuration, traditional SSH brute-force attacks become ineffective, as password authentication is completely disabled and only authorized SSH keys are accepted.

---

## Writing Style

The article was intentionally written in a conversational style to make Linux security more approachable for beginners while still providing practical commands and real-world recommendations.

My goal was not to produce academic documentation, but rather a guide that engineers could immediately apply to secure their own servers.
